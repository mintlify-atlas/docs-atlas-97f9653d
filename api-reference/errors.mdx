## Error Response Format

All API errors follow a consistent JSON structure to help you quickly identify and resolve issues.

### Standard Error Response

```json
{
  "status": "fail",
  "message": "Error description",
  "isOperational": true,
  "stack": "Error stack trace (dev environment only)",
  "data": null
}
```

### Response Fields

| Field | Type | Description |
|-------|------|-------------|
| `status` | string | Always `"fail"` for error responses |
| `message` | string | Human-readable error message |
| `isOperational` | boolean | Indicates if the error is operational (expected) or system error |
| `stack` | string | Stack trace (only included in development environment) |
| `data` | null | Always `null` for error responses |

## HTTP Status Codes

The GameHub API uses standard HTTP status codes to indicate the success or failure of requests.

### Success Codes

| Status Code | Description |
|-------------|-------------|
| `200 OK` | Request succeeded |
| `201 Created` | Resource created successfully |
| `204 No Content` | Request succeeded with no response body |

### Client Error Codes

| Status Code | Description | Common Causes |
|-------------|-------------|---------------|
| `400 Bad Request` | Invalid request parameters | Missing required fields, invalid data format |
| `401 Unauthorized` | Authentication failed | Invalid or expired token, missing credentials |
| `403 Forbidden` | Insufficient permissions | Attempting actions without proper role/permissions |
| `404 Not Found` | Resource not found | Invalid ID, deleted resource |

### Server Error Codes

| Status Code | Description |
|-------------|-------------|
| `500 Internal Server Error` | Unexpected server error |

## Error Handling Best Practices

### Check Status Codes

Always check the HTTP status code before parsing the response body:

```javascript
if (response.status >= 400) {
  const error = await response.json();
  console.error(`Error ${response.status}: ${error.message}`);
}
```

### Handle Token Expiration

Implement automatic token refresh or redirect to login when receiving 401 errors:

```javascript
if (response.status === 401) {
  // Redirect to login or refresh token
  window.location.href = '/login';
}
```

### Display User-Friendly Messages

The `message` field contains descriptive error messages suitable for display to end users:

```javascript
const showError = (error) => {
  alert(error.message);
};
```

### Operational vs System Errors

The `isOperational` field helps distinguish between:

- **Operational errors** (`isOperational: true`): Expected errors like validation failures, resource not found
- **System errors** (`isOperational: false`): Unexpected errors that may require technical investigation

```javascript
if (!error.isOperational) {
  // Log to error monitoring service
  console.error('System error occurred:', error);
}
```

## Development vs Production

In development environments (`NODE_ENV=dev`), error responses include a `stack` field with the full stack trace for debugging. This field is automatically omitted in production for security reasons.

**Development Response:**
```json
{
  "status": "fail",
  "message": "Usuario no encontrado",
  "isOperational": true,
  "stack": "Error: Usuario no encontrado\n    at ...",
  "data": null
}
```

**Production Response:**
```json
{
  "status": "fail",
  "message": "Usuario no encontrado",
  "isOperational": true,
  "data": null
}
```
